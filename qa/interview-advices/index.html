Coding interviews

Practice solving problems. Don’t waste your time solving TopCoder or CodeForces problems (like I did), go straight to LeetCode.com. Many of the good problems on LeetCode are actually unlocked only for premium users (I bought a 1-year subscription when there was a 50% discount on Thanksgiving). You could start solving what is open, and then buy a subscription for a month or more and solve all the locked problems. You could also use my repo as a reference, I checked in most of my solutions.

I recommend starting with Facebook, LinkedIn, Uber, and Airbnb problems on LeetCode, and then switching to Google problems (some of the problems overlap across different companies). Many of the Google problems tend to be ridiculously hard (which can make you feel intimidated) and don’t reflect the level of the problems that are asked nowadays. Also when switching to Google problems, it’s better sorting them in the descending order of IDs, so that the latest added ones showed up first.

Don’t skip Easy problems, some of the “easy” problems tend to be harder than some of the “medium” ones. This is especially true for problems with restrictions like “you need to solve it in O(1) space” or “you cannot use division”. Moreover, the ideas from the easy problems can serve as building blocks for solving harder ones. I personally solved 215 problems by the time I did the onsite interviews (63 easy, 121 medium, 31 hard).

Avoid using an IDE and write directly in the LeetCode editor. It is very similar to coderpad.io and collabedit.com that are used for most of the phone screen interviews.

Invest into a whiteboard (I bought this one), or practice coding on paper before you buy one. Maybe not all of the problems, but about 20%, should be written by hand. You’ll need the skill for Google and Facebook onsite interviews, and also for the system design questions in all the big companies. Writing code on a whiteboard requires having better-formed thoughts before jumping into coding, unlike with an IDE where you can easily erase and move code around. Whiteboard coding also teaches you how to write a more laconic code.

Don’t go too deep into algorithms. You’ll never be asked to write a suffix tree or a balanced tree, or even Dijkstra’s shortest path algorithm. You need to know the following algorithms: sorting (quick, merge, counting, radix -sort algorithms), binary search, DFS, BFS, DSU (might be useful, and it’s a pretty easy data structure), stacks, min/max stack, queues, heaps, linked lists, KMP, hashes (Rabin-Karp). Segment Tree and/or Binary Indexed Tree (aka Fenwick tree) might be useful, and even though you probably won’t be asked to implement them, it’s still good to know that they exist, what operations they allow to perform, and what the runtime complexities of those operations are.

Interviewers try to test your problem solving skills, knowledge of basic data structures and ability to write clean code.

Time yourself, try to stay focused and solve problems as quickly as possible. Interviewers might keep more than one problem in mind, or they might have follow up questions where you end up solving a totally different problem.
